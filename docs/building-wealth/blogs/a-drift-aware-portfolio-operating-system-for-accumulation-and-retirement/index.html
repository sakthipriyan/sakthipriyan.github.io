A Drift-Aware Portfolio Operating System for Accumulation and Retirement

A unified framework that handles buying, selling, liquidity management, and withdrawals under one coherent rule set for both accumulation and retirement phases.

📅 Jun 25, 2026 📖 17 min read

Perpetual Rebalancing: A Drift-Aware Portfolio Operating System for Accumulation and Retirement

1. Purpose

Perpetual Rebalancing is a stateful, drift-aware portfolio operating system designed to manage both:

  • Accumulation: building wealth through ongoing contributions
  • Retirement / Withdrawal: funding spending while preserving portfolio stability

The framework avoids fixed-date rebalancing and instead uses continuous drift control driven by:

  • Forward Water-Filling for deploying fresh capital into underweights
  • Reverse Water-Filling for harvesting from overweights when drift becomes excessive
  • Liquidity Buckets for emergency and retirement spending protection
  • Tax-aware, LTCG-only sell discipline
  • Priority-based capital routing, where liquidity protection takes precedence over discretionary rebalancing

The result is a single portfolio engine that handles buying, selling, liquidity management, and withdrawals under one coherent rule set.

2. Core Design Principles

The system is built around five principles.

2.1 Fresh inflows should do most of the rebalancing

The cheapest and cleanest way to correct portfolio drift is to direct new money into underweight assets rather than selling existing winners. This reduces:

  • turnover
  • taxes
  • unnecessary transaction friction

2.2 Winners should not be constantly cut back to target

Assets are allowed to drift above target within a tolerance band. Sales occur only when portfolio-wide positive drift becomes meaningfully large. This preserves:

  • trend participation
  • lower turnover
  • lower tax realization
  • less whipsaw rebalancing

2.3 Liquidity must be protected before optimization

A portfolio is not only a return engine; it is also a funding system. Therefore:

  • during accumulation, the Emergency Bucket is the first priority if depleted
  • during retirement, the Retirement Bucket is the first priority if below target

This prevents the rebalancing engine from behaving intelligently on paper while failing the actual cashflow needs of the household.

2.4 Sells must be portfolio-level, not ad hoc

When a sell is required, it should not be based on manually picking a winner to trim. Instead:

  • the system computes portfolio-wide positive drift
  • when sell thresholds are breached, it runs a reverse water-filling optimizer
  • the optimizer distributes the sell burden across overweight assets in a mathematically consistent way

2.5 The retirement bucket is a liquidity sink, not a return-seeking allocation sleeve

The retirement bucket exists to:

  • fund withdrawals
  • reduce forced selling of risk assets during market stress
  • absorb harvested gains from overweight assets

It is not part of ordinary drift optimization.

3. System States

The portfolio operates in one of two top-level states:

3.1 Accumulation Mode

The investor is still in the wealth-building phase. Characteristics:

  • regular fresh inflows may continue
  • spending is not funded from the portfolio as a primary source
  • an Emergency Bucket exists for resilience
  • if the emergency bucket is used, it is refilled before normal rebalancing

3.2 Retirement Mode

The portfolio is now responsible for funding living expenses. Characteristics:

  • a Retirement Bucket holds 3 years of expenses
  • portfolio withdrawals are made from the retirement bucket
  • if the retirement bucket is below target, it becomes the first destination for all eligible capital

4. Portfolio Structure

The system is split into two distinct layers.

4.1 Core Portfolio

This is the strategic investment portfolio whose assets participate in ordinary drift optimization. Example sleeves may include:

  • India Equity
  • US Equity
  • Debt
  • Gold
  • Other long-term strategic sleeves

Each core asset has:

  • a target weight
  • a current value
  • a current weight
  • a drift relative to target

The core portfolio participates in:

  • buy-side water-filling
  • sell-side reverse water-filling
  • drift measurement
  • sell trigger evaluation

4.2 Liquidity Buckets

These are special-purpose liquidity reserves and are excluded from ordinary drift optimization logic. There are two bucket types:

A. Emergency Bucket Used during accumulation. Purpose:

  • absorb short-term income disruption / unexpected expense shocks
  • avoid forced liquidation of long-term assets

Rule:

  • if the emergency bucket has been used and is below target, it becomes the first fill destination during accumulation

B. Retirement Bucket Used during retirement. Purpose:

  • hold 3 years of expenses
  • fund withdrawals
  • reduce sequence risk by decoupling spending from market volatility
  • absorb harvested capital from the core portfolio before normal rebalancing resumes

Rule:

  • if below target, it becomes the first fill destination during retirement

5. Liquidity Bucket Design

5.1 Emergency Bucket

The Emergency Bucket exists only in Accumulation Mode.

Purpose

  • short-term resilience
  • funding unexpected spending / temporary income loss
  • keeping the investment engine from being interrupted by short-term shocks

Refill Rule If the emergency bucket is below target because it has been used, then during accumulation:

  • all eligible fresh inflows go to the emergency bucket first
  • harvested capital from sell events, if any, also goes to the emergency bucket first
  • only after the emergency bucket is fully restored does the remaining capital flow to the core portfolio

This makes the emergency bucket the primary liquidity sink during accumulation.

5.2 Retirement Bucket

The Retirement Bucket exists only in Retirement Mode.

Target Size The target is 3 years of expenses. Let:

  • E₁ = next 12 months expected expenses
  • E₂ = following 12 months expected expenses
  • E₃ = third 12 months expected expenses

Then: RB_target = E₁ + E₂ + E₃

In practice, E₂ and E₃ should usually reflect inflation-adjusted projected expenses rather than simply repeating current annual expense.

Allowed Assets The retirement bucket is composed only of:

1. Indian Arbitrage Funds Purpose:

  • INR liquidity sleeve
  • immediate domestic spending reserve
  • low-volatility local cashflow layer

2. USD Treasury Sleeve Purpose:

  • defensive reserve
  • destination for harvested global asset gains
  • low-risk liquid reserve

Treasury Maturity Policy The USD Treasury sleeve is:

  • primarily 0–1 year
  • with some 1–3 year exposure

The bucket is therefore primarily a liquidity and stability reserve, not a duration strategy.

Suggested internal role split A practical interpretation is:

  • near-term spending layer → Indian arbitrage funds
  • outer reserve layer → mostly 0–1 year U.S. Treasuries, with a smaller 1–3 year component

The exact percentage split is a policy choice, but the intent is clear:

  • local spending liquidity in INR
  • reserve stability in short U.S. Treasuries

6. Mathematical Definitions

Assume the core portfolio has n strategic assets. For asset i:

  • Vᵢ = current market value
  • wᵢ = target portfolio weight
  • V = Σ Vᵢ = total core portfolio value
  • pᵢ = Vᵢ / V = current portfolio weight
  • dᵢ = pᵢ - wᵢ = drift of asset i

Interpretation:

  • dᵢ < 0: asset is underweight
  • dᵢ > 0: asset is overweight

6.1 Positive and Negative Drift

Define:

  • dᵢ⁺ = max(dᵢ, 0)
  • dᵢ⁻ = max(-dᵢ, 0)

where:

  • dᵢ⁺ is positive drift magnitude
  • dᵢ⁻ is underweight magnitude

6.2 Total Positive Drift

The system-wide positive drift measure is:

  • D⁺ = Σ dᵢ⁺

This is the aggregate positive drift across the core portfolio.

7. Buy Engine: Forward Water-Filling

The buy engine is used whenever capital is available for deployment into the core portfolio. It is used only after liquidity bucket obligations are satisfied.

7.1 Inputs

The buy engine receives a deployable capital amount C, where C ≥ 0. This capital may come from:

  • fresh SIP / salary inflows
  • dividends / coupons / idle cash
  • residual harvested proceeds after bucket refilling

7.2 Eligible Assets

Only underweight core assets are eligible to receive capital. Define the underweight set:

  • U = { i | dᵢ < 0 }

Overweight assets are ignored by the buy optimizer.

7.3 Buy Objective

The buy engine allocates C across underweight assets to minimize post-allocation drift imbalance, while never selling and never allocating to overweights. Conceptually, it solves:

  • min Σ (dᵢ_post)²

subject to:

  • only underweights receive capital
  • no asset is pushed beyond its target
  • total deployed capital equals available capital C, unless underweights are fully repaired first

This is implemented using bounded water-filling:

  • the deepest deficits receive capital first
  • capital is spread so that deficits are reduced as evenly as possible
  • once an asset reaches target, it stops receiving further capital

8. Sell Engine: Reverse Water-Filling

The sell engine is used only when aggregate positive drift becomes too large. It is the mirror image of the buy engine.

8.1 Sell Trigger

A sell program is activated when:

  • D⁺ > 10%

That is, when total positive drift across the core portfolio exceeds 10%. This is a portfolio-level trigger, not a single-asset trigger.

8.2 Trailing Ceiling State Machine

Once a sell program is activated, the system enters a staged ceiling process.

Ceiling sequence The positive drift ceiling is progressively stepped down:

  • 10% → 9% → 8% → 7% → 6% → 5%

At each execution step, the active ceiling is the current ceiling c. The sell engine only attempts to reduce positive drift down to the active ceiling, not to target and not to zero drift. This preserves:

  • momentum tolerance
  • lower turnover
  • lower tax realization
  • smoother trimming of winners

8.3 Sell Eligibility

Only LTCG-enabled assets are eligible for sale. This means:

  • assets that do not currently have eligible LTCG lots available are excluded from the sell pool
  • the reverse water-filling optimizer only considers assets that are both:
    • above the active ceiling
    • LTCG-eligible

Define the eligible sell set at ceiling c:

  • S(c) = { i | dᵢ > c and asset i has LTCG-eligible sellable lots }

8.4 Excess Drift Above Ceiling

For each eligible seller, define excess drift above the active ceiling:

  • eᵢ(c) = max(dᵢ - c, 0)

Only assets with eᵢ(c) > 0 contribute to the sell optimizer.

8.5 Sell Objective

At a given ceiling c, the sell engine computes sell amounts that minimize the post-sell sum of squared excess drifts above the ceiling. Conceptually, it solves:

  • min Σ (max(dᵢ_post - c, 0))²

subject to:

  • only LTCG-eligible assets may be sold
  • no asset is sold below the active ceiling in that stage
  • sales are bounded by available LTCG-eligible units / lots
  • only the amount required for the current ceiling step is sold

This is implemented using reverse water-filling:

  • the most overweight eligible assets contribute the most
  • sales are distributed so that excess positive drift is flattened as evenly as possible
  • no asset is pushed below the active ceiling during the current step

8.6 Sell Program Progression

A sell program behaves as a state machine.

Stage 1 If D⁺ > 10%, activate sell program at ceiling 10%.

Stage 2 onward Over subsequent review periods, the ceiling may step down:

  • 10%
  • 9%
  • 8%
  • 7%
  • 6%
  • 5%

At each stage:

  • compute eligible sellers above the current ceiling
  • run reverse water-filling
  • harvest post-tax capital
  • route proceeds according to liquidity priority rules
  • stop when the current stage’s objective is satisfied

The program eventually aims to reduce the portfolio to a resting maximum ceiling of 5% rather than to zero drift.

9. Tax and Lot Rules

The system is intentionally tax-aware.

9.1 LTCG-only sell policy

Only lots that qualify as Long-Term Capital Gains are eligible for sale. This prevents the engine from triggering unnecessary short-term tax leakage.

9.2 Net Capital, Not Gross Capital

Whenever a sell occurs, the engine must compute the realizable post-tax proceeds, not merely the gross sale amount. That means:

  • determine which LTCG lots are sold
  • compute gain per lot
  • compute tax liability on the realized gain
  • reserve tax immediately
  • only the remaining net capital becomes deployable capital

So the system never “hallucinates” capital that is actually owed to taxes.

9.3 Sell-lot selection

The exact LTCG lot selection policy is an implementation detail, but the framework requires that:

  • only LTCG-eligible lots are considered
  • tax liability is estimated before proceeds are reused

10. Capital Routing Priority

The heart of the framework is the cash routing cascade. The same optimizer may produce capital, but the destination of that capital depends on portfolio state.

10.1 Accumulation Mode Priority

In accumulation, the priority order is:

Step A — Emergency Bucket first If the emergency bucket is below target because it has been used:

  • all fresh inflows go to the emergency bucket first
  • all harvested net sell proceeds go to the emergency bucket first
  • only after the emergency bucket is fully restored does any remaining capital flow to the core portfolio

Step B — Core portfolio second Once the emergency bucket is fully restored:

  • fresh inflows and residual net sell proceeds are allocated to underweight core assets using forward water-filling

So the accumulation-mode cash priority is: Emergency Bucket → Core Underweights

10.2 Retirement Mode Priority

In retirement, the priority order is:

Step A — Retirement Bucket first If the retirement bucket is below its 3-year target:

  • all fresh inflows go to the retirement bucket first
  • all net harvested proceeds from sell events go to the retirement bucket first
  • only after the retirement bucket is fully restored does any remaining capital flow to the core portfolio

Step B — Core portfolio second Once the retirement bucket is at target:

  • residual capital may be allocated to underweight core assets using forward water-filling

So the retirement-mode cash priority is: Retirement Bucket → Core Underweights

11. Retirement Bucket and Core Portfolio Relationship

The retirement bucket is part of total wealth, but not part of ordinary drift optimization. This distinction is intentional.

The core portfolio Participates in:

  • drift measurement
  • buy optimization
  • sell optimization
  • sell trigger calculations

The retirement bucket Does not participate in:

  • ordinary drift optimization
  • sell trigger calculations
  • reverse water-filling as a normal asset sleeve

Instead it behaves as a priority liquidity sink:

  • if below target, it absorbs capital first
  • if at target, it becomes neutral
  • spending is drawn from it during retirement

This separation prevents the system from immediately “rebalancing away” from the very reserve it is trying to build and protect.

12. Monthly Execution Engine

The system runs as a periodic state machine, typically monthly.

12.1 Step 0 — Update state

At the beginning of the cycle:

  • refresh market values of all core portfolio assets
  • refresh emergency bucket value or retirement bucket value
  • refresh expense assumptions if needed
  • identify whether the portfolio is in Accumulation Mode or Retirement Mode

12.2 Step 1 — Determine active liquidity bucket obligation

Depending on state:

  • If in Accumulation Mode: check whether the Emergency Bucket is below target
  • If in Retirement Mode: check whether the Retirement Bucket is below target

Compute: BucketDeficit = max(0, BucketTarget - BucketCurrent)

12.3 Step 2 — Collect available cash inflows

Aggregate all non-sell cash available for the cycle:

  • fresh SIP / salary inflow
  • dividends
  • coupons
  • idle cash
  • any external contribution

Let this amount be: C_in

12.4 Step 3 — Route inflows to the active liquidity bucket first

If the relevant bucket has a deficit: C_bucket = min(C_in, BucketDeficit)

Transfer C_bucket to the bucket. Then: C_residual = C_in - C_bucket

If the bucket is already full, then C_bucket = 0 and all inflow remains residual.

12.5 Step 4 — If residual cash remains, run forward water-filling

If C_residual > 0, allocate it to underweight core assets using the buy optimizer. This is allowed only after the bucket-first obligation is satisfied for the cycle.

12.6 Step 5 — Compute portfolio drift

Using only the core portfolio, compute for each asset:

  • current weight pᵢ
  • drift dᵢ

Then compute total positive drift: D⁺ = Σ max(dᵢ, 0)

12.7 Step 6 — Evaluate sell trigger

If: D⁺ ≤ 10% then no sell program is triggered this cycle.

If: D⁺ > 10% then activate or continue the sell program using the current active ceiling state.

12.8 Step 7 — Run reverse water-filling sell optimizer

Given active ceiling c:

  • identify all core assets with:
    • dᵢ > c
    • LTCG-eligible lots available
  • compute excess drift above ceiling: eᵢ(c) = max(dᵢ - c, 0)
  • compute sell amounts that minimize post-sell squared excess drift above the ceiling
  • map those sell amounts into actual lot-level sales, constrained to LTCG-eligible units
  • compute gross sale proceeds

12.9 Step 8 — Reserve taxes and compute net harvested capital

For the lots actually sold:

  • compute realized gains
  • compute tax due
  • reserve tax
  • compute net proceeds

Let: C_sell_net be the net harvested capital after tax reservation.

12.10 Step 9 — Route harvested net capital using bucket-first priority

If the active liquidity bucket still has a deficit, route harvested net capital there first.

Allocation to bucket C_bucket_sell = min(C_sell_net, BucketRemainingDeficit) Transfer that amount to the bucket.

Residual harvested capital If any net sell proceeds remain after filling the bucket, they become deployable residual capital.

12.11 Step 10 — If residual harvested capital remains, run forward water-filling again

If the bucket has been fully restored and residual sell proceeds remain, allocate the remainder to underweight core assets using the buy optimizer.

12.12 Step 11 — Update sell program state

If a sell program is active, update its ceiling state. Example:

  • if the 10% stage is complete, next review may proceed to 9%
  • then 8%
  • …
  • until 5%

The exact pacing is a policy choice, but the logical progression is: 10% → 9% → 8% → 7% → 6% → 5%

13. End-to-End Behaviour by Portfolio State

13.1 Accumulation Mode

If emergency bucket is intact

  • collect fresh inflows
  • allocate inflows to underweight core assets
  • if total positive drift > 10%, run sell optimizer
  • route harvested proceeds to underweights

If emergency bucket has been used

  • collect fresh inflows
  • refill emergency bucket first
  • if sell program triggers, harvested proceeds also refill emergency bucket first
  • only after the emergency bucket is restored does capital flow to core underweights

13.2 Retirement Mode

Spending All spending is funded from the retirement bucket.

If retirement bucket is full

  • fresh inflows can go to underweight core assets
  • sell programs, if triggered, harvest from overweight LTCG-eligible core assets
  • harvested proceeds may go to underweights if the bucket remains at target

If retirement bucket is below target

  • all fresh inflows go to the retirement bucket first
  • all harvested net sell proceeds go to the retirement bucket first
  • only after the retirement bucket is fully restored does residual capital flow to underweight core assets

14. Why the Framework Works

The system deliberately separates three different jobs that most portfolios mix together.

14.1 Job 1 — Growth

The core portfolio is allowed to pursue long-term return through strategic assets:

  • India equity
  • US equity
  • debt
  • gold
  • and other long-term sleeves

14.2 Job 2 — Drift control

Drift is controlled through two optimizers:

Buy-side Forward water-filling repairs underweights using fresh capital.

Sell-side Reverse water-filling trims excess positive drift only when portfolio-wide drift becomes too large. This creates a disciplined “buy low / trim high” mechanism without constant churn.

14.3 Job 3 — Liquidity protection

Liquidity needs are handled outside ordinary drift optimization:

  • Emergency Bucket during accumulation
  • Retirement Bucket during retirement

This ensures the portfolio can continue operating through:

  • income shocks
  • emergency spending
  • retirement withdrawals
  • market drawdowns without forcing the drift engine to liquidate strategic assets at the wrong time.

15. Practical Summary of the Full System

Perpetual Rebalancing is best understood as a priority cascade with two symmetric optimizers.

A. Buy side When cash arrives:

  • refill the relevant liquidity bucket first if needed
  • deploy the rest to underweight core assets using forward water-filling

B. Sell side When aggregate positive drift exceeds 10%:

  • activate sell program
  • use reverse water-filling across LTCG-eligible overweight core assets
  • trim only down to the active ceiling
  • reserve tax
  • route net proceeds to the liquidity bucket first if deficient
  • route residual proceeds to underweight core assets

C. State continuity The same framework governs both:

  • accumulation
  • retirement

Only the identity of the priority liquidity bucket changes:

  • Emergency Bucket in accumulation
  • Retirement Bucket in retirement

16. Canonical Rule Set

Below is the framework in one compact list.

  • State: portfolio is either in Accumulation Mode or Retirement Mode
  • Liquidity bucket:
    • Accumulation → Emergency Bucket
    • Retirement → Retirement Bucket
  • Liquidity priority: if the active bucket is below target, it is filled first before core portfolio rebalancing
  • Retirement bucket target: always fill to 3 years of expenses
  • Retirement bucket assets:
    • Indian arbitrage funds
    • USD Treasuries, mostly 0–1 year, with some 1–3 year exposure
  • Core portfolio drift universe: only strategic portfolio assets, excludes emergency / retirement bucket
  • Buy rule: use forward water-filling into underweights only
  • Sell trigger: trigger sell program when total positive drift across core assets exceeds 10%
  • Sell execution:
    • staged ceilings: 10 → 9 → 8 → 7 → 6 → 5
    • use reverse water-filling
    • minimize post-sell sum of squared excess drift above active ceiling
  • Sell eligibility: only assets with LTCG-eligible lots may be sold
  • Tax handling: reserve taxes before capital is reused
  • Spending rule in retirement: all spending comes from retirement bucket

17. Final Interpretation

Perpetual Rebalancing is not a static allocation rule. It is a portfolio operating system with four integrated layers:

  1. A strategic multi-asset portfolio
  2. A drift-aware buy engine
  3. A drift-aware sell engine
  4. A lifecycle liquidity layer for emergency and retirement spending

Its defining feature is that it treats rebalancing, tax-aware selling, emergency liquidity, and retirement withdrawals as one unified system rather than separate problems.
